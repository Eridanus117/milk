# 维护说明

这个仓库当前是一个原生 HTML/CSS/JS 的单页聊天应用，核心设计目标已经从“单一对话页”转成“会话首页 + 系统 6 聊 + 自定义会话”。

## 核心结构

### 系统 6 聊

- 系统聊天固定为 `1 个群聊 + 5 个角色单聊`
- 会话记录仍然落在现有 `sessionList` 和各会话独立的 localforage key 上
- 不改 storage key 的前提下，系统聊天只是在原多会话模型上加了 `systemChatKey / systemChatType / systemRoleId`

用户视角上：

- 首页默认先落到会话列表
- 点进任意系统聊天，再进入具体对话
- 群聊与单聊都有独立最近消息、未读、后台消息配置

### 回复生成

当前有两条消息生成路径：

- 前台即时回复：`simulateReply()`
- 后台离屏回复：`generateSessionReplies()` -> `deliverOffscreenReplies()`

维护原则：

- 两条路径必须共用同一套“按发言者选卡”的内核
- 单聊只能从当前角色专属组或公共组取回复，不能回退到裸 `customReplies` 池随机抽
- 群聊按成员逐条选卡，发言者和字卡池必须一致

当前已经做过一次收口，后续不要再让前台/后台各自维护独立选卡规则。

### 后台消息

后台消息配置已经拆成两类语义：

- `主动发消息`
  - 控制离屏实时投递
  - 只在当前用户切到别的会话时生效
- `恢复`
  - 控制切后台/回前台后的 catchup 补发
  - 当前会话恢复补发时不增加未读

当前调度入口：

- 定时投递：`runBackgroundDeliveryTick()`
- 回前台补发：`runCatchupDelivery()`

维护时不要再把 `allowOffscreenDelivery` 和 `restoreOnReturn` 绑回同一判断里。

### 版本更新

版本可见性和切换基于：

- 页面内嵌 `window.__BUILD_INFO__`
- 远端 `build-info.json`
- 手动/自动检查后判断 `assetVersion` 是否变化

当前更新策略是：

- 发现新版本后提示
- 点击 `立即更新` 时先 `saveData()`
- 然后执行 `location.reload()`

这不是热更新。它是“受控刷新切新版”。

## 最近改造后的稳定边界

当前值得保守对待的区域：

- header 导航与首页入口
- 系统 6 聊的会话标识字段
- 前台/后台共享的回复选卡逻辑
- 后台消息配置的语义映射
- `build-info.json` 与 `index.html` 的版本参数对应关系

如果改这些区域，必须做浏览器级回归，而不只是语法检查。

## 推荐的近端重构

### 安全且高收益

1. 提取统一的 `session delivery` 模块
- 把 `generateSessionReplies / deliverOffscreenReplies / runBackgroundDeliveryTick / runCatchupDelivery` 收到同一块
- 目标是减少 `core.js` 横向膨胀

2. 提取共享的 `reply selection` helper
- 让前台即时回复、群聊回复、后台回复全部只调一个内核
- 避免再次出现角色串卡

3. 收口 header / home / current settings 的 UI 状态控制
- 现在有一部分 DOM 直接改 style，一部分走 class
- 建议统一成更少的状态入口

4. 给关键交互补浏览器回归脚本
- 会话首页
- 当前聊天设置开关
- 版本更新
- 至少保持 1 套移动端视口验证

### 有价值但先别做大的

5. 把构建版本逻辑移到独立文件
- 当前逻辑不复杂，但散在 `core.js` 和 `README`
- 适合收口，不适合大改

## 暂时不建议做的重构

- 重写整个 localforage 存储层
- 一次性把所有全局变量改成模块化状态容器
- 同时重做 UI 导航、数据层、功能抽象

原因：

- 这个仓库刚完成一轮产品形态重排
- 现在更需要“边界收清 + 回归补齐”
- 大重构会把“行为修复”和“结构改造”重新耦合，回到最近这波问题的来源

## 最低回归要求

每次改以下任一部分时，至少做这些验证：

- 改首页/header：
  - 桌面截图
  - 移动端截图
- 改系统 6 聊或回复逻辑：
  - 单聊不串卡
  - 群聊发言人与字卡一致
- 改后台消息：
  - `主动发消息` 生效
  - `恢复` 生效
  - 当前会话恢复不加未读
- 改版本更新：
  - 检测新版本
  - 立即更新后 reload 到新版

## 关于 `claude-personal`

`claude-personal` 适合做这些低风险整理任务：

- 文档起草
- 复盘条目整理
- 重构候选清单归纳

不建议把它放在需要稳定返回的主执行链路里，至少当前环境里它存在长时间不回包的情况。更稳妥的用法是：

- 它先出草稿
- 主链路由本地 agent 审阅、裁剪、落仓
