# 重构 Backlog

按投入产出比排序，先做高收益、低风险的收口型重构。

## 近端安全重构

### 1. 提取统一回复生成内核

- 目标：让前台即时回复、群聊回复、后台离屏回复只走一套按发言者选卡逻辑
- 收益：避免再次出现角色串卡、群聊发言人与字卡错位
- 风险：低

### 2. 提取后台消息模块

- 目标：把 `generateSessionReplies / deliverOffscreenReplies / runBackgroundDeliveryTick / runCatchupDelivery` 从 `core.js` 中收口
- 收益：把“主动发消息 / 恢复”的语义边界固化下来
- 风险：低到中

### 3. 收口首页与 header 状态控制

- 目标：统一 `session-home-active`、按钮高亮、标题显隐、当前设置入口显隐
- 收益：减少 UI 状态散落，降低移动端回归成本
- 风险：低到中

### 4. 抽出版本更新小模块

- 目标：把 `build-info` 读取、状态渲染、立即更新逻辑从大块 `core.js` 中拆出来
- 收益：让更新功能更容易独立测试
- 风险：低

### 5. 把关键浏览器回归脚本沉淀到 `scripts/`

- 目标：在现有冒烟脚本之外，增加更窄但更稳定的专项回归
- 建议优先补：
  - 后台消息开关
  - 版本更新链路
  - 移动端 header / 首页
- 风险：低

## 可以做，但要控制节奏

### 6. 按功能切 `core.js`

- 目标：把首页/导航、后台消息、版本更新、消息生成逐步拆成多个文件
- 前提：每次只切一个功能块，不要一把切完
- 风险：中

### 7. 统一设置项渲染模式

- 目标：减少内联字符串模板 + DOM 手绑混用
- 收益：降低“行为已改但视觉类名没跟着改”的问题
- 风险：中

## 暂时不建议做的大改

### A. 重写整个存储层

- 不建议原因：当前 session/home/system-chat 模型刚稳定，重写容易把正确行为再打散

### B. 一次性全局状态模块化

- 不建议原因：仓库仍以全局状态驱动，直接全面模块化会把“结构改造”和“行为修复”重新耦合

### C. 同时重做 UI 架构 + 数据架构

- 不建议原因：投入大、回归面太宽，现在更应该先把边界和测试稳住
